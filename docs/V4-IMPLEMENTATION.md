# V4 精修优化 — 实施方案文档

## 一、问题分析与总体思路

V4 精修围绕 6 个验收问题展开，其中 BUG-2/BUG-3/OPT-1 都指向同一个核心缺陷：**评分体系的碎片化**。这三个问题的根因是评分逻辑被分散复制到了 3 个不同文件中，各自使用不同的公式，导致数据矛盾。

解决策略：**先统一评分基础设施（scoring.ts），再以此为基础修复所有下游问题（图表、导出、AI 解读）。**

## 二、评分公式设计（BUG-2 + BUG-3 + OPT-1）

### 2.1 设计理念

POC 报告的核心目的是**向客户证明算法相对于人工调度的价值**。因此评分体系应该以"相对改善"为核心逻辑：

- **人工策略 = 基准线 (50分)**：人工策略与自身对比，改善为 0，得分 50。这不是"差"，而是"参照物"。
- **算法策略的得分 = 50 + 改善幅度映射**：算法在各维度的改善程度线性映射到 50 以上。
- **约束遵循独立评分**：约束遵循不做相对比较，而是用扣分制（满分 100，按违反率扣分），因为所有策略都应该遵守约束。

### 2.2 经济性得分 (Economy Score)

**子指标与权重：**
| 子指标 | 权重 | 含义 |
|--------|------|------|
| 里程节降率 | 30% | 更短里程 = 更低油费/过路费 |
| 车次节降率 | 30% | 更少车辆 = 更低调度成本 |
| 时长节降率 | 20% | 更短工时 = 更低人力成本 |
| 满载率提升 | 20% | 更高装载 = 更好的车辆利用 |

**公式：** `sub_score = 50 + improvement_pct × 2.5`（满载率使用 × 3 因为改善幅度通常更小）

**映射效果：**
- 0% 改善 → 50 分（人工基准）
- 4% 改善 → 60 分
- 10% 改善 → 75 分
- 20% 改善 → 100 分（封顶）

**设计理由：** 选择 2.5 作为缩放系数，是因为物流调度 POC 中典型的改善幅度在 5-15% 区间，这个系数让这个区间映射到 62.5-87.5 分，有良好的区分度，同时 20% 以上的改善（非常显著）才会达到满分。

### 2.3 约束遵循得分 (Constraint Score)

**公式：** `100 - avgMaxRate × 1.2 - avgOverallRate × 0.3`

其中：
- `avgMaxRate`：各批次中最高违反率的平均值（反映最严重的约束违反）
- `avgOverallRate`：所有约束类型违反率的总体平均值

**映射效果：**
- 0% 违反率 → 100 分
- 5% 违反率 → ~92 分
- 10% 违反率 → ~85 分
- 20% 违反率 → ~70 分

**设计理由：** 旧公式 `100 - rate × 500` 在 20% 违反率时就变成 0 分，过于敏感。新公式使用双重惩罚机制（最严重违反 + 整体水平），在保留区分度的同时避免分数过快趋零。

### 2.4 合理性得分 (Feasibility Score)

**子指标（等权重）：**
| 子指标 | 含义 |
|--------|------|
| 路线跨度改善率 | 更小跨度 = 更紧凑的路线 |
| 跨区数量改善率 | 更少跨区 = 更低运营复杂度 |
| 绕行率改善率 | 更低绕行 = 更直接的路线 |
| TopK 紧密度改善率 | 更小距离 = 更好的空间聚类 |

**公式：** 与经济性相同的改善率映射模型，`50 + improvement × 2.5`

### 2.5 综合得分

`overall = economy × 0.4 + constraint × 0.3 + feasibility × 0.3`

权重通过 `ScoringWeights` 接口可配置（已在 `types/index.ts` 中定义）。

### 2.6 单策略场景处理

当没有人工基准时（如 r5 单策略报告），使用绝对基准评分：
- 经济性：基于满载率的绝对水平
- 合理性：基于路线指标的经验阈值

## 三、HTML 导出重构（BUG-1）

### 3.1 方案选择

**选择了"数据驱动的完整 ECharts 渲染"方案（方向 A 的优化实现）。**

三个方案的对比：

| 方案 | 优点 | 缺点 |
|------|------|------|
| A. 完整重写 HTML 模板 | 完全交互、视觉一致 | 工作量大 |
| B. 截图式（base64 图片） | 实现简单 | 失去交互性、文件大 |
| C. 混合方案 | 平衡 | 复杂度高、不一致 |

**实际实现是 A 的高效版本：** 在 TypeScript 中构建所有图表的 ECharts option 对象（复用数据处理逻辑），序列化为 JSON 嵌入 HTML，由浏览器端 ECharts 渲染。

### 3.2 实现架构

```
TypeScript 构建阶段:
  buildChartDefs() → ChartDef[] (id, title, section, option, insight)
      ↓
  JSON.stringify → chartOptionsJson
      ↓
HTML 渲染阶段:
  echarts.init(dom).setOption(option) for each chart
```

### 3.3 满足的约束

1. ✅ 自包含单文件（CSS/JS/数据内联，ECharts 通过 CDN）
2. ✅ 包含所有已渲染图表（基于 DataAvailability 动态决定）
3. ✅ 包含每张图表的 AI 解读
4. ✅ 包含 KPI 概览卡片
5. ✅ 包含报告元信息
6. ✅ 合理文件大小（HTML 模板约 30-80KB，远小于 2MB）
7. ✅ ECharts CDN 加载，无网络时图表不渲染但文本内容可读

### 3.4 零违反场景的 HTML 导出

零违反报告（如 r7）的约束遵循章节导出为全宽总结卡片，与页面展示一致。

## 四、零违反视觉优化（OPT-2）

### 4.1 问题

旧实现使用半宽 `ChartCard` + `EmptyState`，在 `md:grid-cols-2` 网格中只占一列，右侧留白。

### 4.2 解决方案

替换为全宽（`md:col-span-2`）的自定义卡片，包含：
- 醒目的绿色圆形勾选图标 + "全部约束检查通过"标题
- 根据 `DataAvailability` 动态列出所有已检查的约束类型，每项带绿色勾选标记
- 底部显示 AI 解读文本

## 五、AI 解读质量提升（OPT-3）

### 5.1 修复的函数

- `genS1`：现在引用统一评分函数计算的具体分数，包括各维度得分和综合排名
- `genS2`：现在引用排名第一的策略名称、综合得分和与第二名的分差
- `genF7`：现在引用各策略的具体路线跨度、绕行率、跨区数量和聚类紧密度数据

### 5.2 数据一致性

`genS1` 和 `genS2` 调用 `calculateAllStrategyScores()`，确保解读中引用的分数与 S1 雷达图和 S2 排名图中展示的完全一致。
