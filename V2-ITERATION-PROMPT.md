## 一、迭代背景与核心变更

### 1.1 为什么需要重构指标体系

V1 版本的指标分类沿用了 TuningAgent 的内部分类体系（Type A 边界类 / Type B 形态类 / Type C 综合效能）。但 TuningAgent 的分类是面向**算法调参工程师**的内部视角，而 POC 报告的受众是**客户决策层和业务管理者**。两者的认知框架存在根本差异。

客户关心的核心问题是三个：

1. **经济性**：算法能不能帮我省钱？（车次数、里程、时长、成本）
2. **约束遵循**：算法的方案是否违反了运营规则？（超时、超载、超里程）
3. **合理性**：算法的路线在实际中跑得通吗？（跨区、绕行、路线跨度）

这三个维度构成了一个完整的评价闭环：**在不违反约束（维度2）、不损害落地可行性（维度3）的前提下，尽量实现经济性优化（维度1）**。这也是实施顾问在 POC 阶段向客户推荐算法策略的核心逻辑。

### 1.2 变更范围

本次迭代的变更范围如下：

| 变更项 | V1 状态 | V2 目标 |
|--------|---------|---------|
| 指标分类体系 | Type A/B/C（算法工程师视角） | 经济性/约束遵循/合理性（业务决策视角） |
| 图表数量与类型 | 16 张固定图表 | 弹性图表体系，按数据可用性动态渲染 |
| 数据模型 | 固定字段 | 扩展约束违反和合理性字段 |
| KPI 概览 | 5 项固定 KPI | 按数据可用性动态展示 |
| 浮动导航 | KPI / Type A / Type B / Type C | KPI / 经济性 / 约束遵循 / 合理性 |
| AI 解读 | 通用解读 | 分维度解读，聚焦业务决策价值 |

## 二、新指标分类体系详解

### 2.1 维度一：经济性指标（Economic Efficiency）

> 核心问题：算法能不能帮客户省钱？

这是 POC 报告中最直观、最有说服力的部分。经济性指标直接反映调度方案的成本效益。

**指标清单**（并非所有指标都一定有数据，按数据可用性动态展示）：

| 指标 | 字段名 | 单位 | 说明 | 数据来源 |
|------|--------|------|------|---------|
| 车次数 | `vehicleCount` | 次 | 总派车次数，直接影响固定成本 | 必有 |
| 总里程 | `totalDistance` | km | 所有车次的行驶里程之和 | 必有 |
| 总工作时长 | `totalDuration` | min | 所有车次的工作时长之和（含装卸等待） | 必有 |
| 平均满载率 | `avgLoadRate` | % | 所有车次的加权平均满载率 | 必有 |
| 分车型平均满载率 | `avgLoadRateByType` | % | 按车型分组的平均满载率 | 有多车型时 |
| 分车型使用次数 | `vehicleCountByType` | 次 | 各车型的使用频次，不同车型成本不同 | 有多车型时 |
| 总成本 | `totalCost` | 元 | 基于计费结构计算的总成本 | 有计费结构时 |
| 里程节降率 | `distanceSavingsRate` | % | (人工里程 - 算法里程) / 人工里程 × 100% | 计算得出 |

**关键设计原则**：
- 不是所有项目都有全部指标。例如，有些项目只有一种车型，那么"分车型"相关的图表就不应出现。
- 有些项目没有给出计费结构，那么"总成本"就不展示。
- 图表组件应当根据数据可用性**动态决定是否渲染**，而非显示空图表。

### 2.2 维度二：约束遵循情况（Constraint Compliance）

> 核心问题：算法的方案是否违反了运营规则？

路径规划是一个实际业务问题。在现实中，即使是人工调度也不一定能严格遵循所有约束条件。因此，这个维度的核心不是"是否完美遵循"，而是**对比各策略的约束违反程度**，帮助客户理解算法在约束遵循方面的表现。

**约束类型清单**（同样按数据可用性动态展示）：

| 约束类型 | 字段名 | 阈值来源 | 违反判定 | 说明 |
|----------|--------|---------|---------|------|
| 单车工作时长上限 | `maxDurationLimit` | 项目配置 | duration > limit | 超时意味着司机疲劳风险 |
| 单车行驶里程上限 | `maxDistanceLimit` | 项目配置 | distance > limit | 超里程意味着成本失控 |
| 装载重量上限 | `maxWeightLimit` | 车型配置 | weight > limit | 超重违反交规 |
| 装载体积上限 | `maxVolumeLimit` | 车型配置 | volume > limit | 超方影响安全 |
| 装载件数上限 | `maxQtyLimit` | 车型配置 | qty > limit | 超件影响装卸效率 |
| 跨区数量上限 | `maxCrossRegionLimit` | 项目配置 | crossRegion > limit | 跨区过多影响司机熟悉度 |
| 卸货点数上限 | `maxStopLimit` | 项目配置 | stopCount > limit | 点数过多影响时效 |

**展示逻辑**：
- 对于每种约束，统计各策略下**违反该约束的车次数量和违反比例**。
- 同时展示**违反程度**（超出阈值的幅度分布）。
- 即使人工调度也可能违反约束，这恰恰是对比的价值所在——如果算法的违反率低于人工，说明算法在约束遵循方面更优。

### 2.3 维度三：合理性指标（Operational Feasibility）

> 核心问题：算法的路线在实际中跑得通吗？

合理性指标衡量的是调度方案的"落地可行性"。一个方案即使经济性很好、也没有违反硬约束，但如果路线不合理（比如跨越整个城市来回绕行），在实际运营中也是不可接受的。

**指标清单**：

| 指标 | 字段名 | 单位 | 说明 | 为什么重要 |
|------|--------|------|------|-----------|
| 路线跨度 | `routeSpan` | km | 路线外接圆直径 | 跨度过大意味着路线覆盖范围不合理 |
| 跨区数量 | `crossRegionCount` | 个 | 路线穿越的区域围栏数量 | 这里的"区"是 C-ROS 系统中的区域围栏（GIS Fence），而非行政区划。跨区多意味着司机需要在不同区域间穿梭 |
| 路线折角与绕行 | `detourRatio` | % | 实际路线长度 / 理论最短路线长度 | 绕行率高意味着路线规划不够紧凑 |
| 卸货点数 | `stopCount` | 个 | 单车次的卸货站点数量 | 点数过多影响配送时效和司机体验 |
| 聚类紧密度 (Top-K) | `topKAvg` | km | 每个站点到其 K 个最近邻站点的平均距离 | 值越小说明站点越集中，路线越紧凑 |
| 路线交叉重叠 | `routeOverlap` | 次 | 不同车次路线之间的交叉次数 | 交叉多意味着区域划分不合理 |
| 点间距分布 | `stopInterval` | km | 相邻卸货点之间的距离 | 间距过大意味着路线中存在"跳跃" |

**关键说明**：
- "区域围栏"是 C-ROS 系统中的概念，由实施顾问在系统中配置。它通常对应客户的配送分区（如"浦东区"、"松江区"），但不一定与行政区划完全重合。
- 合理性指标与约束遵循有一定重叠（如跨区数量既是约束也是合理性指标），但视角不同：约束遵循关注"是否违反了设定的上限"，合理性关注"分布是否合理"。

### 2.4 三个维度的关系

```
┌─────────────────────────────────────────────────┐
│                  POC 评价框架                      │
│                                                   │
│   ┌──────────┐   ┌──────────┐   ┌──────────┐    │
│   │  经济性   │   │ 约束遵循  │   │  合理性   │    │
│   │          │   │          │   │          │    │
│   │ 省多少钱？│   │ 违规多少？│   │ 跑得通吗？│    │
│   │          │   │          │   │          │    │
│   │ 车次/里程 │   │ 超时/超载 │   │ 跨度/绕行 │    │
```

## 三、图表体系重构

### 3.1 设计原则

1. **数据驱动渲染**：图表不再是固定的 16 张。每个维度下的图表根据数据可用性动态决定是否渲染。如果某个指标没有数据，对应的图表卡片不显示（而非显示空图表）。
2. **图表类型服务于对比逻辑**：选择图表类型时，首先考虑"这张图要回答什么问题"，然后选择最适合回答该问题的图表类型。
3. **批次粒度 vs 汇总粒度**：部分图表展示各批次的逐一对比（如各批次车次数），部分图表将所有批次的数据汇总后对比（如满载率分布）。这取决于图表要回答的问题。
4. **多策略扩展性**：所有图表必须支持 n 组策略同时对比（最多 8 组），series 和 legend 动态生成。

### 3.2 维度一：经济性图表

| 编号 | 图表名称 | 图表类型 | 粒度 | 对比逻辑 | 渲染条件 |
|------|---------|---------|------|---------|---------|
| E1 | 各批次车次数对比 | 分组柱状图 | 按批次 | X=批次, Y=车次数, 每组一个策略 | 必有 |
| E2 | 各批次总里程对比 | 分组柱状图 | 按批次 | X=批次, Y=总里程(km) | 必有 |
| E3 | 各批次总工作时长对比 | 分组柱状图 | 按批次 | X=批次, Y=总时长(min) | 必有 |
| E4 | 各批次平均满载率对比 | 折线图(带标记点) | 按批次 | X=批次, Y=满载率(%), 折线趋势对比 | 必有 |
| E5 | 分车型平均满载率 | 分组柱状图(嵌套) | 按车型×策略 | X=车型, 分组=策略, Y=满载率(%) | 有多车型时 |
| E6 | 分车型使用次数 | 堆叠柱状图 | 按策略 | X=策略, 堆叠=车型, Y=车次数 | 有多车型时 |
| E7 | 总成本对比 | 水平柱状图 | 汇总 | Y=策略, X=总成本(元), 按成本排序 | 有计费结构时 |
| E8 | 里程节降率趋势 | 面积折线图 | 按批次 | X=批次, Y=节降率(%), 含目标区间虚线 | 有多批次时 |

**E1 详细规格**：
- 图表类型：分组柱状图（grouped bar chart）
- X 轴：订单批次名称
- Y 轴：车次数
- 每个批次下并排显示各策略的柱子
- 人工策略的柱子使用琥珀橙色，算法策略使用各自的颜色
- Tooltip 显示：批次名、策略名、车次数、与人工的差值

**E4 详细规格**：
- 图表类型：折线图，每条线代表一个策略
- 标记点（markPoint）标注每条线的最高值和最低值
- 人工策略使用虚线，算法策略使用实线
- 区域填充（areaStyle）使用半透明色

**E8 详细规格**：
- 面积折线图，Y 轴为节降率百分比
- 用两条水平虚线标注目标区间（如 12%-18%），虚线之间填充浅色区域
- 如果只有 1 个批次，此图不渲染

### 3.3 维度二：约束遵循图表

| 编号 | 图表名称 | 图表类型 | 粒度 | 对比逻辑 | 渲染条件 |
|------|---------|---------|------|---------|---------|
| C1 | 约束违反率总览 | 分组柱状图 | 按约束类型×策略 | X=约束类型, 分组=策略, Y=违反比例(%) | 至少有1种约束有数据 |
| C2 | 工作时长超限分布 | 箱线图 | 按策略 | 仅展示超限车次的超限幅度分布 | 有时长上限时 |
| C3 | 行驶里程超限分布 | 箱线图 | 按策略 | 仅展示超限车次的超限幅度分布 | 有里程上限时 |
| C4 | 装载量超限热力图 | 热力图 | 按策略×装载维度 | X=装载维度(重量/体积/件数), Y=策略, 色值=违反比例 | 有装载上限时 |
| C5 | 约束违反车次明细 | 散点图(气泡) | 按车次 | X=超限幅度, Y=约束类型, 气泡大小=影响程度 | 至少有1种约束有数据 |

**C1 详细规格**：
- 这是约束遵循维度的**核心总览图**
- X 轴：约束类型名称（工作时长、行驶里程、装载重量、装载体积、跨区数量、卸货点数等）
- Y 轴：违反比例（违反车次数 / 总车次数 × 100%）
- 每个约束类型下并排显示各策略的柱子
- 柱子上方标注具体的违反车次数（如 "3/22"）
- 如果某种约束所有策略的违反率都为 0，该约束类型不显示（避免无意义的空柱子）
- 关键：人工调度也可能有约束违反，这是对比的价值所在

**C4 详细规格**：
- 热力图，X 轴为装载维度（重量、体积、件数、托盘），Y 轴为策略名称
- 色值从绿色（0% 违反）到红色（高违反率）
- 鼠标悬停显示具体的违反车次数和违反比例
- 如果项目只涉及单一装载维度（如只有重量），退化为简单的分组柱状图

### 3.4 维度三：合理性图表

| 编号 | 图表名称 | 图表类型 | 粒度 | 对比逻辑 | 渲染条件 |
|------|---------|---------|------|---------|---------|
| F1 | 路线跨度分布 | 箱线图(含散点) | 按策略 | 各策略的路线跨度分布对比 | 有跨度数据时 |
| F2 | 跨区数量分布 | 堆叠柱状图 | 按策略 | X=策略, 堆叠=跨区数(0/1/2/3+), Y=车次数 | 有跨区数据时 |
| F3 | 绕行率分布 | 密度曲线图(KDE) | 汇总 | 各策略的绕行率密度曲线叠加对比 | 有绕行数据时 |
| F4 | 卸货点数分布 | 箱线图(含散点) | 按策略 | 各策略的单车卸货点数分布对比 | 必有 |
| F5 | 聚类紧密度 (Top-K) | 散点图 | 按车次 | X=车次编号, Y=Top-K均值, 颜色=策略 | 有Top-K数据时 |
| F6 | 点间距累积分布 | CDF 曲线 | 汇总 | 各策略的点间距CDF曲线叠加对比 | 有点间距数据时 |
| F7 | 多维合理性雷达图 | 雷达图 | 汇总 | 6维归一化指标（跨度/跨区/绕行/点数/Top-K/点间距） | 至少3个维度有数据 |

**F1 详细规格**：
- 箱线图，每个策略一组箱线
- 叠加散点（jitter scatter），展示每个车次的具体值
- 箱线图的颜色使用策略调色板
- 在图表上方用虚线标注"合理跨度参考线"（如果项目有配置）

**F2 详细规格**：
- 堆叠柱状图，X 轴为策略名称
- 堆叠层为跨区数量等级：0区（仅本区）、1区、2区、3区及以上
- 颜色从浅到深表示跨区数量递增
- Tooltip 显示各等级的车次数和占比

**F7 详细规格**：
- 雷达图，各维度归一化到 0-100 分
- 归一化逻辑：对于"越小越好"的指标（如跨度、跨区数），取反后归一化
- 每个策略一条雷达线
- 如果可用维度不足 3 个，此图不渲染

### 3.5 综合评估图表（跨维度）

除了三个维度各自的图表外，增加一组跨维度的综合评估图表：

| 编号 | 图表名称 | 图表类型 | 说明 | 渲染条件 |
|------|---------|---------|------|---------|
| S1 | 策略综合评分雷达图 | 雷达图 | 三大维度（经济性/约束遵循/合理性）的综合评分 | 必有 |
| S2 | 策略排名对比 | 水平柱状图 | 各策略的加权综合评分排名 | 必有 |

**S1 详细规格**：
- 三个维度各自计算一个 0-100 的综合分数
- 经济性得分：基于里程节降率、车次节约率、满载率提升的加权平均
- 约束遵循得分：100 - 加权违反率（违反率越低得分越高）
- 合理性得分：基于跨度、跨区、绕行等指标的归一化加权平均
- 每个策略一条雷达线，三个顶点分别代表三个维度

## 四、数据模型更新

### 4.1 扩展 VehicleDetail 接口

在 V1 的 `VehicleDetail` 基础上，增加约束遵循和合理性相关字段：

```typescript
interface VehicleDetail {
  // === V1 已有字段 ===
  vehicleId: string;
  vehicleType: string;
  orderCount: number;
  stopCount: number;
  distance: number;           // 行驶距离(km)
  duration: number;           // 工作时长(min)
  loadRate: number;           // 满载率(%)
  
  // === V2 新增：装载明细 ===
  weightLoad: number;         // 实际装载重量(kg)
  volumeLoad: number;         // 实际装载体积(m³)
  qtyLoad: number;            // 实际装载件数
  palletLoad?: number;        // 实际装载托盘数（可选）
  
  // === V2 新增：约束相关 ===
  durationOverLimit?: number;     // 超出时长上限的分钟数（null=未超限）
  distanceOverLimit?: number;     // 超出里程上限的km数
  weightOverLimit?: number;       // 超出重量上限的kg数
  volumeOverLimit?: number;       // 超出体积上限的m³数
  qtyOverLimit?: number;          // 超出件数上限的件数
  crossRegionOverLimit?: number;  // 超出跨区上限的个数
  stopOverLimit?: number;         // 超出点数上限的个数
  
  // === V2 新增：合理性相关 ===
  routeSpan: number;          // 路线跨度(km) - 外接圆直径
  crossRegionCount: number;   // 跨区数量
```

## 五、Mock 数据更新

### 5.1 Mock 数据生成原则

V2 的 Mock 数据需要体现以下特征：

1. **人工策略的"经验型"特征**：
   - 满载率通常较高（人工调度员凭经验装满车）
   - 但里程和时长不一定最优（人工倾向于按区域习惯排线，不一定是最短路径）
   - 约束违反率可能不为零（人工有时会"灵活处理"，比如稍微超时）
   - 合理性指标通常较好（人工排线符合司机习惯）

2. **算法策略的"优化型"特征**：
   - "里程优先"策略：里程最短，但可能跨区较多、路线跨度较大
   - "均衡优化"策略：各项指标均衡，没有明显短板
   - "时效优先"策略：工作时长最短，但里程可能不是最优
   - "满载优先"策略：满载率最高，但可能导致部分车次绕行

3. **约束违反的真实感**：
   - 不是所有策略都完美遵循约束
   - 人工策略可能有 5-10% 的时长超限（人工"灵活处理"）
   - 里程优先策略可能有少量跨区超限
   - 满载优先策略可能有少量装载超限

4. **数据可用性配置**：
   - Mock 一个"完整数据"场景（所有指标都有数据）
   - 在 `DataAvailability` 中配置为大部分为 `true`
   - 但 `hasCostStructure` 设为 `false`（模拟没有计费结构的常见场景）
   - `hasPalletLimit` 设为 `false`（模拟不涉及托盘的场景）
